## 1. 方案概述

个人全能站点采用 **「主基座 + 微前端子应用 + 统一 API」** 架构：

- **Next.js 14** 承担全站外壳：顶栏导航、暗黑模式、404、前台 AI 浮窗、路由壳层。
- **Qiankun 2** 加载两个 **Vite + React 18** 子应用：`app-web`（访客站）、`app-admin`（管理后台）。
- **NestJS 10 + Prisma 6 + SQLite** 提供 REST API、RBAC、CMS、AI 向量检索与操作审计。

与早期「每页一个 Web Component / 六个独立子应用」方案相比，当前实现：

| 维度 | 当前方案 |
|------|---------|
| 微前端框架 | Qiankun（`registerMicroApps` + `vite-plugin-qiankun`） |
| 子应用数量 | 2 个（前台聚合六页 + 管理端） |
| 子应用技术 | React + React Router Data API + Tailwind |
| 静态资源路径 | `/micro/web/`、`/micro/admin/` |
| 管理端 | 完整 Admin UI + JWT + RBAC + AI 管理 |

## 2. 技术栈总览

### 2.1 前端

| 层级 | 技术 | 说明 |
|------|------|------|
| 主基座 | Next.js 14 App Router | `(site)/[[...slug]]`、`admin/[[...slug]]` |

| 搜索 | fuse.js | 基座全站搜索 |
| Markdown | react-markdown | 博客正文（子应用内） |
| 图表 | echarts + echarts-for-react | AI 回复中的图表块（基座渲染） |

### 2.2 后端

| 类别 | 技术 |
|------|------|
| 框架 | NestJS 10 |
| ORM | Prisma 6 |
| 数据库 | SQLite（开发 `dev.db`，生产 `prod.db`） |
| 认证 | Passport JWT |
| 密码 | bcryptjs |
| 日志 | nestjs-pino + 自研 `AppLogService` 写库 |
| AI | LangChain + LangGraph + OpenAI 兼容 API |
| 向量库 | LanceDB（本地目录，与 SQLite 同卷持久化） |

### 2.3 运维

| 进程并发 | concurrently（本地 `dev:all`） |

```
│  app-web (React)          │  app-admin (React)                   │
│  首页/关于/博客/项目/联系/友链 │  CMS / RBAC / AI管理 / 日志        │
└────────────┬──────────────┴──────────────┬──────────────────────┘
             │                              │
             └──────────────┬───────────────┘
                            ▼
              ┌─────────────────────────┐
              │  nest-server :3001       │
              │  /api/* 公开 + 管理鉴权   │
              └────────────┬────────────┘
                           ▼
              ┌─────────────────────────┐
              │  SQLite + LanceDB        │
              └─────────────────────────┘
```

### 3.1 请求路径（生产）

Nginx 统一入口：

- `/` → Next.js（页面壳 + Qiankun 挂载）
- `/api/*` → NestJS
- `/micro/web/`、`/micro/admin/` → 子应用静态资源（**entry 必须带尾斜杠**）

## 5. 微前端方案（Qiankun）

### 5.1 为何选 Qiankun

- 子应用保持 **完整 React 工程**（Router、Hooks、Ant Design 等），比原生 Web Components 更适合复杂后台。
- 前台六页合并为 **一个子应用**，减少 entry 数量与构建体积。
- 与 Vite 官方生态通过 `vite-plugin-qiankun` 对接，支持独立运行（`dev:standalone`）。

### 5.2 注册方式（基座）

`next-host/utils/qiankun.ts` 在客户端动态 `import('qiankun')`，避免 SSR 报错：

```typescript
// 两个子应用，同一容器 #micro-container
{
  name: 'app-web',
  entry: NEXT_PUBLIC_MICRO_WEB,  // 默认 http://localhost:4001
  container: '#micro-container',
  activeRule: (loc) => isPublicSitePath(loc.pathname),
  props: { theme, apiBase, onMessageSuccess? },
},
{
  name: 'app-admin',
  entry: NEXT_PUBLIC_MICRO_ADMIN, // 默认 http://localhost:4007
  container: '#micro-container',
  activeRule: (loc) => isAdminPath(loc.pathname),
  props: { theme, apiBase },
}
```

`start({ prefetch: true, singular: false, sandbox: false })`：`sandbox: false` 降低与 Vite/React 的兼容成本。

### 5.3 子应用生命周期

`qiankun-subapps/_shared/mountApp.tsx` 封装：

- `mount`：`createRoot` 挂载到 props.container 或 `#root`
- `unmount`：精确 `root.unmount()`，避免切换子应用时 DOM 残留

`app-web/src/main.tsx` 使用 `renderWithQiankun`；非 Qiankun 环境走 `getStandaloneHostProps()` 独立启动。

### 5.4 生产 HTML 修复

`fixQiankunHtml.mjs` 将 `index.html` 内联 `import()` 改为 `<script type="module" src="...">`，详见 [部署文档 §7.6](./部署文档.md#76-fixqiankunhtmlmjs)。

### 5.5 与「六个子应用」历史方案对比

| 项目 | 旧方案（文档 v1） | 当前 v2 |
|------|------------------|---------|
| 子应用数 | 6（home/about/…） | 2（web/admin） |
| 微前端 | 每页独立 entry | 单 app-web 内 React Router |
| 构建 | 6 次 Vite build | 2 次 + nginx 静态目录 |

## 6. 主基座 next-host

### 6.1 路由

| 文件 | 作用 |
|------|------|
| `app/(site)/[[...slug]]/page.tsx` | 前台占位页，真实 UI 在 Qiankun 容器 |
| `app/admin/[[...slug]]/page.tsx` | 管理后台占位 |
| `router/siteRoutes.ts` | `SITE_ROUTE_CONFIG`：驱动导航、activeRule、404 |

前台路由 key 与 `app-web` 的 `WEB_ROUTE_CONFIG` 对齐：`home | about | blog | projects | contact | links`。

### 6.2 布局与能力

- `(site)/layout.tsx`：`MainNav` + `QiankunProvider` + 页脚
- `admin/layout.tsx`：`AdminQiankunProvider`（管理端挂载逻辑）
- 全局：`Providers`（Theme）、站点配置 `useSiteConfig`（`/api/site/config`）
- **AI 浮窗**：`components/ai-assistant/*`，SSE 消费 `/api/ai/chat`

### 6.3 环境变量

| 变量 | 开发默认 | 生产 |
|------|---------|------|
| `NEXT_PUBLIC_API_BASE` | `http://localhost:3001/api` | `{ORIGIN}/api` |
| `NEXT_PUBLIC_MICRO_WEB` | `http://localhost:4001` | `{ORIGIN}/micro/web/` |
| `NEXT_PUBLIC_MICRO_ADMIN` | `http://localhost:4007` | `{ORIGIN}/micro/admin/` |

## 7. 子应用 qiankun-subapps

### 7.1 app-web（前台）

**端口**：4001（`registerSubAppDevPort(4001)`）

**路由**（`src/router/webRoutes.ts`）：

| path | 页面 |
|------|------|
| `/` (index) | HomePage |
| `about` | AboutPage |
| `blog`、`blog/:id` | BlogPage |
| `projects` | ProjectsPage |
| `contact` | ContactPage |
| `links` | LinksPage |

**数据**：通过 `ApiBaseProvider` 使用基座传入的 `apiBase` 请求公开 API。

**联系页**：基座可通过 `window.__HOST_CALLBACKS__.onMessageSuccess` 接收留言成功提示（`QiankunProvider` props getter）。

### 7.2 app-admin（管理后台）

**端口**：4007

**路由**：React Router + `pageRegistry` / `routes.ts`，basename 为 `/admin`（独立运行时为 `/`）。

**功能模块**（与 `rbac-module-tree.ts` 路径对齐）：

- 站点配置：站点设置、导航、关于、联系
- 内容：文章、项目、友链、留言
- 系统：模块/权限/角色/用户、操作日志、应用日志
- AI：数据配置、知识库分块、同步记录

**鉴权**：`utils/auth.ts` 存 JWT；`PermissionGuard` 按权限码控制按钮；请求头带 `Authorization: Bearer`。

### 7.3 _shared

- `mountApp.tsx`：Qiankun 挂载/卸载
- `components/`：部分页面级 UI 复用
- `api.ts`：fetch 封装
- `fixQiankunHtml.mjs`：生产构建后处理

## 8. 后端 nest-server

### 8.1 模块划分

| 模块 | 职责 |
|------|------|
| `ArticleModule` | 文章 CRUD（公开 list/detail + admin） |
| `ProjectModule` | 项目 |
| `MessageModule` | 留言 |
| `LinkModule` | 友链 |
| `SiteModule` | `SiteConfig` 单行配置（nav/about/contact JSON） |
| `AuthModule` | `POST /api/auth/login` → JWT |
| `AdminModule` | CMS 管理接口，`/api/admin/*` |
| `RbacModule` | 模块树、角色、用户、权限树 |
| `AiModule` | 公开 SSE 聊天 + 管理端配置/同步/向量 |
| `LoggingModule` | `AppLog` 写入、管理端日志查询 |

### 8.2 公开 API 前缀

| 前缀 | 示例 |
|------|------|
| `/api/site` | `GET config` |
| `/api/article` | `list`、`:id`、`slug/:slug` |
| `/api/project` | `list` |
| `/api/message` | `POST` 创建留言 |
| `/api/link` | `list` |
| `/api/ai` | `POST chat`（SSE）、`DELETE chat/session` |
| `/api/auth` | `POST login` |

### 8.3 管理 API

- **CMS**：`AdminController` `@Controller('api/admin')`，守卫 `JwtAuthGuard` + `PermissionsGuard`，权限码如 `admin:articles:view`。
- **RBAC**：`RbacController` 同前缀下 `rbac/modules`、`rbac/roles`、`rbac/users`、`auth/profile`。
- **AI 管理**：`AdminAiController` `@Controller('api/admin/ai')` — 配置、同步、知识库分块列表/删除。
- **日志**：`AdminLogsController` — 应用日志、审计日志查询。

### 8.4 横切能力

- **ValidationPipe**：`whitelist` + `transform`
- **CORS**：localhost:3000/4001/4007 + `CORS_ORIGIN`
- **监听**：`0.0.0.0:3001`（容器内可被 Nginx 访问）

## 10. 认证与 RBAC

### 10.1 登录流

1. `POST /api/auth/login` `{ username, password }`
2. 返回 JWT（payload 含用户 id）
3. 子应用存储 token，请求 `/api/admin/*` 带头

### 10.2 权限模型

- **模块** `AdminModule`：菜单树，`code` 与前端页面目录对应
- **权限点** `AdminPermission`：`menu` / `button` / `api`，`code` 如 `admin:articles:create`
- **角色** `AdminRole`：`isSuper` 超管跳过权限校验
- **同步**：`rbac-module-tree.ts` 定义标准树；`syncRbacModules` 部署时增量补菜单

### 10.3 守卫

```typescript
@UseGuards(JwtAuthGuard, PermissionsGuard)
@RequirePermissions('admin:articles:view')
```

前端 `PermissionGuard` 与按钮级权限码一致，避免 UI 与 API 不一致。

## 11. AI 小助手

### 11.1 能力

| 能力 | 实现要点 |
|------|---------|
| 前台问答 | `AiChatService.streamChat` → SSE：`session` / `token` / `tool_*` / `done` / `error` |
| 会话记忆 | LangGraph `MemorySaver` + `sessionId` |
| 知识库 | 文章/项目/留言/AppLog 加载器 → 分块 → Embedding → LanceDB |
| 工具调用 | 如 `generate-chart` 生成图表数据，基座 `ChartBlock` 渲染 |
| 配置 | `AiConfig` 表优先；管理端修改后 `resetTable` 向量 |

### 11.2 配置优先级

```
管理后台 AiConfig（DB）
    ↓ 覆盖
环境变量 OPENAI_*（.env / docker-compose）
    ↓ 未配置则
AI 功能不可用或降级提示
```

### 11.3 管理端流程

1. 「数据配置管理」填写 Key、Base URL、模型、向量维度  
2. 「一键同步数据源」→ `AiIngestService` 写 LanceDB + `AiSyncRecord`  
3. 前台浮窗提问 → RAG 检索 + 流式回答  

### 11.4 向量存储

- 路径：`LANCE_DB_PATH`（默认 `./data/lancedb` / 生产 `/app/data/lancedb`）
- 与 SQLite 同属 Docker 卷 `docker_api-data`
- 镜像使用 `node:20-slim`（glibc），避免 Alpine 下 LanceDB native 加载失败

## 12. 通信与状态同步

### 12.1 基座 → 子应用 props

| 字段 | 说明 |
|------|------|
| `theme` | `light` / `dark`，注册时与 `initGlobalState` 一致 |
| `apiBase` | 如 `http://localhost:3001/api` |
| `onMessageSuccess` | 仅 app-web 联系页（可选） |

### 12.2 Qiankun 全局状态

`initGlobalState({ theme })` → 子应用 `onGlobalStateChange` 监听主题变更（`syncQiankunTheme`）。

### 12.3 路由协同

- Next 负责浏览器 URL（如 `/blog/1`）
- 子应用 React Router **basename `/`**，路径与 Next 一致
- `activeRule` 由基座 `isPublicSitePath` / `isAdminPath` 判定，避免错误加载子应用

### 12.4 加载态隔离

`QiankunProvider` 将 loading 遮罩与 `#micro-container` 分离，避免 `setReady` 导致 re-render 清空已挂载子应用 DOM。

## 13. 本地开发与构建

### 13.1 安装

```bash
pnpm run install:all
cd nest-server && pnpm exec prisma generate && pnpm exec prisma migrate dev && pnpm exec ts-node prisma/seed.ts
```

### 13.2 启动

```bash
pnpm run dev:all   # api:3001 + web:3000 + app-web:4001 + app-admin:4007
```

浏览器：**http://localhost:3000**  
管理：**http://localhost:3000/admin** 或 `pnpm run dev:admin:standalone` → :4007

### 13.3 独立调试子应用

| 命令 | 访问 |
|------|------|
| `dev:web:standalone` | http://localhost:4001 |
| `dev:admin:standalone` | http://localhost:4007 |

### 13.4 生产构建

```bash
pnpm run build:subapps   # app-web + app-admin dist
pnpm run build:web       # 先子应用再 next build
```

Docker 生产构建由 `Dockerfile.nginx` / `Dockerfile.web` 在镜像内完成，无需本地先 build。

### 13.5 常用维护脚本

```bash
pnpm run db:migrate
pnpm run db:seed
pnpm --dir nest-server run prisma:sync-rbac
```

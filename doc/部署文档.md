## 1. 架构概览

本项目采用 **Next.js 主基座 + Qiankun 微前端（2 个子应用）+ NestJS 后端** 架构。生产环境通过 **3 个 Docker 容器** 对外提供统一入口（默认 80 端口）。

```
                         用户浏览器
                              │
                              ▼
                    ┌─────────────────┐
                    │  Nginx (80)     │  docker-nginx-1
                    │  统一入口        │
                    └────────┬────────┘
                             │
           ┌─────────────────┼─────────────────┐
           │                 │                 │
           ▼                 ▼                 ▼
    /api/* 代理        /micro/* 静态      /  页面路由
           │                 │                 │
           ▼                 ▼                 ▼
   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
   │ NestJS API   │  │ app-web      │  │ Next.js 主站 │
   │ (3001)       │  │ app-admin    │  │ (3000)       │
   │ docker-api   │  │ 静态 dist    │  │ docker-web   │
   └──────┬───────┘  └──────────────┘  └──────────────┘
          │
          ├─ SQLite：/app/data/prod.db（卷 docker_api-data）
          └─ LanceDB：/app/data/lancedb（AI 向量，同卷）
```

### 各组件职责

| 组件 | 技术栈 | 端口（容器内） | 职责 |
|------|--------|---------------|------|
| **nginx** | Nginx 1.27 | 80 | 反向代理、子应用静态资源托管 |
| **web** | Next.js 14 | 3000 | 主基座：布局、导航、路由、Qiankun 注册、AI 浮窗 |
| **api** | NestJS 10 + Prisma 6 | 3001 | REST API：CMS、RBAC、AI、日志、站点配置 |
| **app-web** | Vite + React + Qiankun | — | 前台六页（首页/关于/博客/项目/联系/友链） |
| **app-admin** | Vite + React + Qiankun | — | 管理后台（CMS、RBAC、AI 管理、日志） |

### 子应用与路径映射

| 子应用 | 开发端口 | 容器内静态目录 | 公网 Entry URL |
|--------|---------|---------------|----------------|
| **app-web** | 4001 | `/usr/share/nginx/micro/web/` | `{ORIGIN}/micro/web/` |
| **app-admin** | 4007 | `/usr/share/nginx/micro/admin/` | `{ORIGIN}/micro/admin/` |

> **注意**：Entry URL **必须带尾部斜杠** `/`。无斜杠时 Nginx 会 301 到带斜杠地址；若误配为 Next 路由则可能返回 404 HTML，导致 Qiankun `LOADING_SOURCE_CODE` 失败。

### 前台路由（由 app-web 承载）

| 路径 | 页面 |
|------|------|
| `/` | 首页 |
| `/about` | 关于我 |
| `/blog`、`/blog/:id` | 博客列表/详情 |
| `/projects` | 作品集 |
| `/contact` | 联系/留言 |
| `/links` | 友链 |

管理后台基座路径：`/admin/*`（由 app-admin 承载）。

## 4. 环境变量说明

生产变量写在服务器 **`/opt/personal-site/.env`**，由 `docker compose --env-file .env` 读取。

### 4.1 根目录 `.env`（最小示例）

```env
# 站点公网地址（无尾部斜杠）
PUBLIC_ORIGIN=http://47.116.30.137

# Nginx 映射到宿主机的端口
HTTP_PORT=80
```

### 4.2 变量如何生效

| 变量 | 影响范围 | 说明 |
|------|---------|------|
| `PUBLIC_ORIGIN` | web 镜像 **构建时** + api **运行时** CORS | 推导 `NEXT_PUBLIC_API_BASE`、`NEXT_PUBLIC_MICRO_WEB`、`NEXT_PUBLIC_MICRO_ADMIN` |
| `HTTP_PORT` | nginx 端口映射 | 默认 80 |

### 4.3 构建时注入的前端变量（Dockerfile.web）

在 **构建 web 镜像时** 写入 Next.js 产物，改 `PUBLIC_ORIGIN` 后需 **重建 web**：

```env
NEXT_PUBLIC_API_BASE=${PUBLIC_ORIGIN}/api
NEXT_PUBLIC_MICRO_WEB=${PUBLIC_ORIGIN}/micro/web/
NEXT_PUBLIC_MICRO_ADMIN=${PUBLIC_ORIGIN}/micro/admin/
```

本地开发对应 `next-host/.env.local`（见 `.env.local.example`）。

### 4.4 后端运行时变量（api 容器）

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `DATABASE_URL` | `file:/app/data/prod.db` | SQLite（compose 内写死） |
| `CORS_ORIGIN` | `{PUBLIC_ORIGIN}` | 生产跨域来源 |
| `JWT_SECRET` | `personal-site-prod-secret` | **生产务必修改** |
| `ADMIN_PASSWORD` | `admin123` | seed 时管理员密码 |
| `NODE_OPTIONS` | web: `--max-old-space-size=384` | 轻量机防 OOM（见 compose） |

### 4.5 AI 小助手（api 容器，可选）

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `OPENAI_API_KEY` | — | 未配置则前台 AI 不可用 |
| `OPENAI_BASE_URL` | `https://api.openai.com/v1` | 兼容 DeepSeek、通义等 |
| `OPENAI_CHAT_MODEL` | `gpt-4o-mini` | 对话模型 |
| `OPENAI_EMBEDDING_MODEL` | `text-embedding-3-small` | 向量化模型 |
| `LANCE_DB_PATH` | `/app/data/lancedb` | 与 SQLite 同卷持久化 |
| `AI_CHUNK_SIZE` | `800` | 文档分块 |
| `AI_CHUNK_OVERLAP` | `100` | 分块重叠 |

**DB 优先于 env**：管理后台「AI小助手管理 → 数据配置管理」可在线改 Key/模型；更新配置会重置向量表，需重新同步数据源。

### 4.6 首次部署与已有 `.env`

`docker/deploy.sh` 仅在服务器 **不存在** `.env` 时写入 `PUBLIC_ORIGIN` 与 `HTTP_PORT`，避免覆盖已配置的 `JWT_SECRET`、`OPENAI_API_KEY` 等。首次部署后建议在服务器手动补全敏感变量并 `docker compose up -d` 重建 api。

## 5. 一键部署（推荐）

脚本：`docker/deploy.sh` → 打包上传 → `docker/remote-deploy.sh` 在服务器构建并启动。

### 5.1 密码部署

```bash
chmod +x docker/deploy.sh
./docker/deploy.sh 47.116.30.137 '你的SSH密码'
```

### 5.2 密钥部署

```bash
./docker/deploy.sh 47.116.30.137
```

### 5.3 自定义 SSH 用户

```bash
SSH_USER=ubuntu ./docker/deploy.sh 你的IP
```

### 5.4 脚本流程

```
本地 deploy.sh
  1. tar 打包（排除 node_modules、.next、dist、*.db、.git）
  2. scp → /opt/personal-site/
  3. 无 .env 时写入 PUBLIC_ORIGIN、HTTP_PORT
  4. 解压并 chmod remote-deploy.sh

服务器 remote-deploy.sh
  5. 清理 next-host 遗留旧路由目录（避免 Next 构建冲突）
  6. docker compose down（不带 -v，保留 docker_api-data）
  7. docker compose up -d --build
  8. docker/wait-healthy.sh 等待首页/API 就绪
  9. exec api node dist/prisma/seed.js（空表才写演示数据）
```

### 5.5 耗时参考

| 阶段 | 预计 |
|------|------|
| 上传 | 10–30 秒 |
| 构建 api | 1–3 分钟 |
| 构建 web（Next） | 2–5 分钟 |
| 构建 nginx（2 个子应用） | 3–8 分钟 |
| **合计** | **约 8–20 分钟**（视机器与网络） |

### 5.6 部署成功标志

```bash
docker compose -f /opt/personal-site/docker/docker-compose.yml ps
# nginx / web / api 均为 Up，web、api 为 healthy

curl -I http://你的IP/
curl http://你的IP/api/article/list
curl -I http://你的IP/micro/web/
curl -I http://你的IP/micro/admin/
```

管理后台：`http://你的IP/admin`（默认账号 `admin`，密码见 `ADMIN_PASSWORD`）。

## 6. 手动部署（逐步操作）

### 6.1 上传代码

```bash
cd /path/to/个人网站
COPYFILE_DISABLE=1 tar czf deploy.tar.gz \
  --exclude='node_modules' \
  --exclude='.next' \
  --exclude='dist' \
  --exclude='*.db' \
  --exclude='*.db-journal' \
  --exclude='.git' \
  .
scp deploy.tar.gz root@你的IP:/opt/personal-site/
```

### 6.2 配置 `.env`

```bash
ssh root@你的IP
cd /opt/personal-site
tar -xzf deploy.tar.gz  # 若使用 tar 包
cat > .env << 'EOF'
PUBLIC_ORIGIN=http://你的公网IP或域名
HTTP_PORT=80
JWT_SECRET=请替换为随机长字符串
ADMIN_PASSWORD=请替换为强密码
EOF
```

### 6.3 构建并启动

```bash
chmod +x docker/remote-deploy.sh
./docker/remote-deploy.sh
```

或：

```bash
docker compose -f docker/docker-compose.yml --env-file .env up -d --build
chmod +x docker/wait-healthy.sh
./docker/wait-healthy.sh
docker compose -f docker/docker-compose.yml exec -T api node dist/prisma/seed.js
```

### 6.4 seed 行为说明

- 管理员 `admin`：`upsert`，密码随 `ADMIN_PASSWORD` 更新。
- RBAC：模块树为空时初始化；`syncRbacModules` 增量补新菜单权限。
- 文章/项目/留言/友链：**仅对应表 count=0 时** 写入演示数据，**重复部署不删生产内容**。

## 7. Docker 服务详解

### 7.1 目录结构

```
个人网站/
├── docker/
│   ├── docker-compose.yml   # 三容器编排
│   ├── Dockerfile.api       # NestJS + Prisma + LanceDB
│   ├── Dockerfile.web       # Next.js 主基座
│   ├── Dockerfile.nginx     # Nginx + 子应用构建
│   ├── nginx.conf           # 路由规则
│   ├── deploy.sh            # 本地一键部署
│   ├── remote-deploy.sh     # 服务器构建启动
│   └── wait-healthy.sh      # 健康等待
├── nest-server/
├── next-host/
├── qiankun-subapps/
│   ├── _shared/             # 共享组件、fixQiankunHtml.mjs
│   ├── app-web/
│   └── app-admin/
└── .env                     # 服务器生产配置（不提交 Git）
```

### 7.2 compose 服务

```yaml
services:
  nginx:   # 对外 ${HTTP_PORT:-80}，依赖 web + api
  web:     # Next.js :3000，仅内网
  api:     # NestJS :3001，卷 api-data → /app/data
volumes:
  api-data:
    name: docker_api-data   # 固定卷名，防止 compose 项目名变化丢数据
```

### 7.3 Dockerfile.api

1. `node:20-slim`（LanceDB 需 glibc，不用 Alpine）
2. `prisma generate` + `nest build` + 编译 `seed.ts`
3. 启动：`prisma migrate deploy` → `node dist/src/main.js`

### 7.4 Dockerfile.web

1. 构建参数 `PUBLIC_ORIGIN` 推导 `NEXT_PUBLIC_*`
2. 复制 `qiankun-subapps/_shared`（样式 token 引用）
3. `next build` → `pnpm start`

### 7.5 Dockerfile.nginx

1. 构建 `app-web`：`VITE_BASE=/micro/web/` + `fixQiankunHtml.mjs`
2. 构建 `app-admin`：`VITE_BASE=/micro/admin/` + `fixQiankunHtml.mjs`
3. 复制 `nginx.conf` 与 dist 到运行镜像

### 7.6 fixQiankunHtml.mjs

`vite-plugin-qiankun` 会在 `index.html` 注入内联 `import('/micro/.../static/xxx.js')`。Qiankun `import-html-entry` 经 eval 执行时无法跑 ESM `import()`，会报 `ReadableStream` / `LOADING_SOURCE_CODE` 错误。构建后脚本改为动态插入 `<script type="module" src="...">`。

## 8. 路由与 URL 规划

### 8.1 Nginx 规则（nginx.conf）

| 请求路径 | 处理 | 示例 |
|---------|------|------|
| `/api/*` | 代理 → api:3001 | `/api/article/list` |
| `/micro/web/` | 静态 alias | app-web entry |
| `/micro/admin/` | 静态 alias | app-admin entry |
| `/micro/web`、`/micro/admin`（无斜杠） | **301** → 带斜杠 | Qiankun 兼容 |
| `/*` | 代理 → web:3000 | `/`、`/admin`、`/blog/1` |

### 8.2 公开 API（节选）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/site/config` | 站点名、导航、关于、联系配置 |
| GET | `/api/article/list` | 文章列表 |
| GET | `/api/article/:id` | 文章详情 |
| GET | `/api/article/slug/:slug` | slug 查询 |
| GET | `/api/project/list` | 项目列表 |
| GET | `/api/link/list` | 友链列表 |
| POST | `/api/message` | 提交留言 |
| POST | `/api/ai/chat` | AI 流式问答（SSE） |
| POST | `/api/auth/login` | 管理端登录 |

管理端 CMS / RBAC / AI 管理接口前缀：`/api/admin/*`（JWT + 权限点）。

### 8.3 验证

```bash
curl -I http://你的IP/
curl http://你的IP/api/site/config
curl -I http://你的IP/micro/web/
curl -I http://你的IP/micro/web    # 应 301
```

## 9. 域名与 HTTPS 配置

### 9.1 修改 PUBLIC_ORIGIN

```env
PUBLIC_ORIGIN=https://yourdomain.com
```

**必须重建 web**（`NEXT_PUBLIC_*` 在构建期固化）：

```bash
cd /opt/personal-site
docker compose -f docker/docker-compose.yml --env-file .env build --no-cache web
docker compose -f docker/docker-compose.yml --env-file .env up -d web nginx api
```

子应用 entry 随 nginx 镜像中的 `VITE_BASE` 构建，改域名后通常需 **重建 nginx**：

```bash
docker compose -f docker/docker-compose.yml --env-file .env build nginx
docker compose -f docker/docker-compose.yml --env-file .env up -d nginx
```

### 9.2 Certbot + Nginx（思路）

在宿主机或侧车申请证书，扩展 `docker/nginx.conf` 增加 443 与证书路径，80 跳转 443。

### 9.3 CDN / SLB 终止 HTTPS

回源到服务器 80，`PUBLIC_ORIGIN` 设为 `https://域名`。

## 10. 日常运维命令

工作目录：`/opt/personal-site`。

```bash
COMPOSE="docker compose -f docker/docker-compose.yml --env-file .env"

# 状态
$COMPOSE ps

# 日志
$COMPOSE logs -f
$COMPOSE logs -f api

# 重启
$COMPOSE restart
$COMPOSE restart api

# 停止（保留卷）
$COMPOSE down

# 启动（不构建）
$COMPOSE up -d

# 进入容器
$COMPOSE exec api sh
$COMPOSE exec nginx sh

# 迁移
$COMPOSE exec api pnpm exec prisma migrate deploy

# 清理 Docker 缓存（磁盘紧张时）
docker system prune -a
```

### 按变更范围重建

| 变更内容 | 重建目标 |
|---------|---------|
| nest-server | `api` |
| next-host | `web` |
| qiankun-subapps、nginx.conf | `nginx` |
| PUBLIC_ORIGIN / 域名 | `web` + `nginx`（必要时 `api` 仅 CORS） |

## 12. 故障排查

### 12.1 无法访问

| 检查项 | 操作 |
|--------|------|
| 容器状态 | `$COMPOSE ps` |
| 防火墙 80 | 云控制台 |
| 端口监听 | `ss -tlnp \| grep :80` |
| Nginx 日志 | `$COMPOSE logs nginx` |

### 12.2 502 Bad Gateway

- 刚启动：等待 `wait-healthy` 或 web/api `healthy`。
- web OOM：查看 `logs web`，考虑 swap 或调大 `NODE_OPTIONS`。
- api 迁移失败：`logs api`，检查 `/app/data/prod.db` 权限。

```bash
curl -I http://127.0.0.1/projects
curl -I http://127.0.0.1/api/article/list
$COMPOSE restart web api nginx
```

### 12.3 子应用 LOADING_SOURCE_CODE / ReadableStream

```bash
curl -s http://你的IP/micro/web/ | head -5
# 正确：含 document.createElement('script'); s.type='module'
# 错误：内联 import('/micro/web/static/...')
```

解决：确认 `Dockerfile.nginx` 执行了 `fixQiankunHtml.mjs`，重建 nginx。

### 12.4 子应用加载到 Next 404 页

- Entry 无尾部斜杠或 `NEXT_PUBLIC_MICRO_*` 配错。
- `curl -I http://IP/micro/web` 应 301；`curl -sL .../micro/web/` 的 title 应为子应用而非主站。

### 12.5 「页面加载中...」不消失

1. 浏览器 Console 看 Qiankun 报错  
2. Network 检查 `/micro/web/` 与 JS 是否 200  
3. `curl http://IP/api/article/list`

### 12.6 构建 OOM

```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile && sudo mkswap /swapfile && sudo swapon /swapfile
$COMPOSE build
```

### 12.7 Next 构建路径冲突

报错 `two parallel pages that resolve to the same path`：服务器残留旧 `next-host/app/about` 等。`remote-deploy.sh` 已清理；可手动：

```bash
rm -rf next-host/app/about next-host/app/blog next-host/app/contact \
       next-host/app/links next-host/app/projects
rm -f next-host/app/page.tsx
$COMPOSE up -d --build web
```

### 12.8 compose 路径错误

始终使用：


## 13. 数据库查看与备份

### 13.1 数据位置

| 环境 | SQLite | LanceDB |
|------|--------|---------|
| 本地 | `nest-server/dev.db` | `nest-server/data/lancedb` |
| 生产 | 卷 `docker_api-data` → 容器 `/app/data/prod.db` | `/app/data/lancedb` |

### 13.2 备份

```bash
docker cp $($COMPOSE ps -q api):/app/data/prod.db ./backup-$(date +%Y%m%d).db
```

### 13.3 查看（服务器）

```bash
$COMPOSE exec api ls -la /app/data/
$COMPOSE exec api sh -c "node -e \"
const { PrismaClient } = require('@prisma/client');
const p = new PrismaClient();
p.article.findMany({ take: 3 }).then(console.log).finally(() => p.\\\$disconnect());
\""
```

### 13.4 本地 Prisma Studio

```bash
cd nest-server
pnpm exec prisma studio
```

生产库可先 `docker cp` 到本机，再 `DATABASE_URL="file:../prod-snapshot.db" pnpm exec prisma studio`。

> **勿对生产库执行 `prisma migrate dev`**，服务器仅 `migrate deploy`。

## 14. 安全建议

1. SSH 密钥登录，关闭密码登录（可选）  
2. `.env`、`*.db` 不提交 Git  
3. 修改 `JWT_SECRET`、`ADMIN_PASSWORD`  
4. 限制 22 端口来源 IP  
5. 定期备份 `prod.db`（含 `AiConfig` 密钥）  
6. 启用 HTTPS  
7. 管理端强密码 + RBAC 最小权限

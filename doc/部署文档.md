## 1. 架构概览

本项目采用 **Next.js 主基座 + Qiankun 微前端子应用 + NestJS 后端** 架构。生产环境通过 **3 个 Docker 容器** 对外提供统一入口（80 端口）。

```
                         用户浏览器
                              │
                              ▼
                    ┌─────────────────┐
                    │  Nginx (80)     │  docker-nginx
                    │  统一入口        │
                    └────────┬────────┘
                             │
           ┌─────────────────┼─────────────────┐
           │                 │                 │
           ▼                 ▼                 ▼
    /api/* 代理        /micro/* 静态      /  页面路由
           │                 │                 │
           ▼                 ▼                 ▼
   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
   │ NestJS API   │  │ 6 个子应用    │  │ Next.js 主站 │
   │ (3001)       │  │ dist 静态文件 │  │ (3000)       │
   │ docker-api   │  │ 内嵌 nginx   │  │ docker-web   │
   └──────┬───────┘  └──────────────┘  └──────────────┘
          │
          ▼
   ┌──────────────┐
   │ SQLite 数据库 │
   │ (Docker Volume)│
   └──────────────┘
```

### 各组件职责

| 组件 | 技术栈 | 端口（容器内） | 职责 |
|------|--------|---------------|------|
| **nginx** | Nginx 1.27 | 80 | 反向代理、子应用静态资源托管 |
| **web** | Next.js 14 | 3000 | 主基座：导航、布局、路由、Qiankun 注册 |
| **api** | NestJS 10 + Prisma | 3001 | REST API：文章、项目、留言、友链 |
| **子应用 ×6** | Vite + React + Qiankun | — | 首页/关于/博客/项目/联系/友链 |

### 子应用与路径映射

| 子应用 | 容器内静态目录 | 公网 Entry URL |
|--------|--------------|----------------|
| app-home | `/usr/share/nginx/micro/home/` | `{ORIGIN}/micro/home/` |
| app-about | `/usr/share/nginx/micro/about/` | `{ORIGIN}/micro/about/` |
| app-blog | `/usr/share/nginx/micro/blog/` | `{ORIGIN}/micro/blog/` |
| app-projects | `/usr/share/nginx/micro/projects/` | `{ORIGIN}/micro/projects/` |
| app-contact | `/usr/share/nginx/micro/contact/` | `{ORIGIN}/micro/contact/` |
| app-links | `/usr/share/nginx/micro/links/` | `{ORIGIN}/micro/links/` |

> **注意**：Entry URL **必须带尾部斜杠** `/`，否则 Nginx 无法匹配子应用路由，请求会落入 Next.js 返回 404 页面，导致 Qiankun 加载失败。

## 3. 部署前准备

### 3.1 阿里云防火墙配置

1. 登录 [阿里云轻量应用服务器控制台](https://swas.console.aliyun.com/)
2. 进入实例 → **防火墙** 标签页
3. 添加入站规则：

| 协议 | 端口 | 来源 | 说明 |
|------|------|------|------|
| TCP | 80 | 0.0.0.0/0 | HTTP 访问 |
| TCP | 443 | 0.0.0.0/0 | HTTPS（配置域名后） |
| TCP | 22 | 你的 IP | SSH（建议限制来源 IP） |

### 3.2 确认 Docker 可用

SSH 登录服务器后执行：

```bash
docker --version
docker compose version
```

若未安装，参考 [Docker 官方安装文档](https://docs.docker.com/engine/install/) 安装。

### 3.3 创建部署目录

```bash
mkdir -p /opt/personal-site
```

### 3.4 （推荐）配置 SSH 密钥登录

```bash
# 本地生成密钥（若还没有）
ssh-keygen -t ed25519 -C "deploy@personal-site"

# 上传公钥到服务器
ssh-copy-id root@你的服务器IP
```

配置密钥后，部署脚本可省略密码参数，更安全。

## 4. 环境变量说明

生产环境变量写在项目根目录的 `.env` 文件中，由 `docker compose --env-file .env` 读取。

### 4.1 根目录 `.env`（生产部署用）

```env
# 站点公网访问地址（无尾部斜杠）
# 使用 IP：http://47.116.30.137
# 使用域名：https://yourdomain.com
PUBLIC_ORIGIN=http://47.116.30.137

# Nginx 对外暴露端口
HTTP_PORT=80
```

### 4.2 变量如何生效

| 变量 | 影响范围 | 说明 |
|------|---------|------|
| `PUBLIC_ORIGIN` | web 镜像构建 + api 运行时 | 决定 `NEXT_PUBLIC_*` 系列变量的默认值 |
| `HTTP_PORT` | nginx 容器端口映射 | 默认 80 |

### 4.3 构建时注入的前端变量（Dockerfile.web）

以下变量在 **构建 web 镜像时** 写入 Next.js 产物，**运行时无法修改**，变更后需重新构建 web 容器：

```env
NEXT_PUBLIC_API_BASE=http://47.116.30.137/api
NEXT_PUBLIC_MICRO_HOME=http://47.116.30.137/micro/home/
NEXT_PUBLIC_MICRO_ABOUT=http://47.116.30.137/micro/about/
NEXT_PUBLIC_MICRO_BLOG=http://47.116.30.137/micro/blog/
NEXT_PUBLIC_MICRO_PROJECTS=http://47.116.30.137/micro/projects/
NEXT_PUBLIC_MICRO_CONTACT=http://47.116.30.137/micro/contact/
NEXT_PUBLIC_MICRO_LINKS=http://47.116.30.137/micro/links/
```

> 以上地址由 `PUBLIC_ORIGIN` 自动推导，一般无需手动设置。若需自定义，可在 `docker-compose.yml` 的 `web.build.args` 中覆盖。

### 4.4 后端运行时变量（api 容器）

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `DATABASE_URL` | `file:/app/data/prod.db` | SQLite 数据库路径 |
| `CORS_ORIGIN` | `{PUBLIC_ORIGIN}` | 允许跨域的来源 |
| `NODE_ENV` | `production` | 运行环境 |
| `JWT_SECRET` | — | 管理后台 JWT 密钥 |
| `ADMIN_PASSWORD` | — | 管理员初始密码（seed） |

### 4.5 AI 小助手变量（api 容器，可选）

启用前台 AI 问答浮窗与后台向量同步需在 `.env` 中配置 OpenAI 兼容 API：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `OPENAI_API_KEY` | — | LLM / Embedding API Key（必填才启用 AI） |
| `OPENAI_BASE_URL` | `https://api.openai.com/v1` | 兼容 DeepSeek、通义等 |
| `OPENAI_CHAT_MODEL` | `gpt-4o-mini` | 对话模型 |
| `OPENAI_EMBEDDING_MODEL` | `text-embedding-3-small` | 向量化模型 |
| `LANCE_DB_PATH` | `/app/data/lancedb` | LanceDB 向量库目录（挂载在 api-data 卷内） |
| `AI_CHUNK_SIZE` | `800` | 文档分块大小 |
| `AI_CHUNK_OVERLAP` | `100` | 分块重叠长度 |

部署后进入管理后台 **AI小助手管理 → 数据配置管理**，选择数据源（博客/项目/留言/系统日志）执行「一键同步数据源」，向量化完成后前台右下角 AI 浮窗即可基于知识库回答。

## 5. 一键部署（推荐）

项目提供了自动化部署脚本 `docker/deploy.sh`，完成：打包 → 上传 → 写环境变量 → Docker 构建 → 启动 → 初始化种子数据。

### 5.1 使用密码部署

```bash
# 在项目根目录执行
chmod +x docker/deploy.sh

./docker/deploy.sh 47.116.30.137 '你的SSH密码'
```

### 5.2 使用 SSH 密钥部署（推荐）

```bash
# 已配置 ssh-copy-id 后，省略密码参数
./docker/deploy.sh 47.116.30.137
```

### 5.3 自定义 SSH 用户

```bash
SSH_USER=ubuntu ./docker/deploy.sh 你的IP
```

### 5.4 脚本执行流程

```
1. tar 打包项目（排除 node_modules、.next、dist、*.db、.git）
2. scp 上传到服务器 /opt/personal-site/
3. 写入 .env（PUBLIC_ORIGIN、HTTP_PORT）
4. docker compose up -d --build（构建 3 个镜像并启动，数据库卷 docker_api-data 保留）
5. 执行 prisma seed（**仅空表写入演示数据，已有文章/项目/留言不会被清空**）
6. 输出容器状态与访问地址
```

### 5.5 首次构建耗时参考

| 阶段 | 预计耗时 |
|------|---------|
| 上传代码 | 10–30 秒 |
| 构建 web（Next.js） | 1–2 分钟 |
| 构建 api（NestJS） | 30–60 秒 |
| 构建 nginx（6 个子应用） | 2–4 分钟 |
| **合计** | **约 5–10 分钟** |

### 5.6 部署成功标志

```bash
# 容器均为 Up 状态
docker compose -f /opt/personal-site/docker/docker-compose.yml ps

# 浏览器访问返回 200
curl -I http://你的IP/
curl http://你的IP/api/article/list
curl -I http://你的IP/micro/home/
```

## 6. 手动部署（逐步操作）

若不想使用脚本，可按以下步骤在服务器上手动操作。

### 6.1 上传代码

**方式 A：本地打包上传**

```bash
# 本地
cd /path/to/个人网站
tar czf deploy.tar.gz \
  --exclude='node_modules' \
  --exclude='.next' \
  --exclude='dist' \
  --exclude='*.db' \
  --exclude='.git' \
  .

scp deploy.tar.gz root@你的IP:/opt/personal-site/
```

**方式 B：Git 克隆（若代码在远程仓库）**

```bash
# 服务器
cd /opt
git clone 你的仓库地址 personal-site
```

### 6.2 配置环境变量

```bash
# 服务器
cd /opt/personal-site

cat > .env << 'EOF'
PUBLIC_ORIGIN=http://你的公网IP或域名
HTTP_PORT=80
EOF
```

### 6.3 构建并启动

```bash
cd /opt/personal-site

# 构建所有镜像并后台启动
docker compose -f docker/docker-compose.yml --env-file .env up -d --build
```

### 6.4 初始化数据库

```bash
# 首次部署执行，写入演示数据（文章、项目、友链等）
# 已有数据时 seed 不会清空，可安全重复执行
docker compose -f docker/docker-compose.yml exec -T api node dist/prisma/seed.js
```

> 注意：旧版 seed 会先 `deleteMany` 再写入；当前版本仅在对应表为空时插入演示数据，**重复部署不会丢失生产内容**。

### 6.5 验证

```bash
docker compose -f docker/docker-compose.yml ps
docker compose -f docker/docker-compose.yml logs --tail=50
```

## 7. Docker 服务详解

### 7.1 目录结构

```
个人网站/
├── docker/
│   ├── docker-compose.yml    # 编排文件
│   ├── Dockerfile.api        # NestJS 后端镜像
│   ├── Dockerfile.web        # Next.js 主站镜像
│   ├── Dockerfile.nginx      # Nginx + 子应用静态资源镜像
│   ├── nginx.conf            # Nginx 路由配置
│   └── deploy.sh             # 一键部署脚本
├── nest-server/              # 后端源码
├── next-host/                # 主基座源码
├── qiankun-subapps/          # 6 个子应用
│   └── _shared/
│       └── fixQiankunHtml.mjs  # 子应用 HTML 构建后修复脚本
├── .env                      # 生产环境变量（部署时生成）
└── .dockerignore             # Docker 构建排除规则
```

### 7.2 docker-compose.yml 服务说明

```yaml
services:
  nginx:   # 对外 80 端口，依赖 web + api
  web:     # Next.js，内部 3000，不对外暴露
  api:     # NestJS，内部 3001，不对外暴露
volumes:
  api-data:  # SQLite 数据库持久化卷
networks:
  app-net:   # 容器间通信网络
```

### 7.3 各 Dockerfile 构建流程

#### Dockerfile.api

```
1. 安装 nest-server 依赖（pnpm）
2. prisma generate + nest build
3. 编译 seed.ts
4. 运行时：prisma migrate deploy → node dist/src/main.js
5. 数据库持久化在 /app/data/prod.db（Docker Volume）
```

#### Dockerfile.web

```
1. 注入 NEXT_PUBLIC_* 环境变量（构建参数 PUBLIC_ORIGIN）
2. 安装 next-host 依赖
3. next build
4. 运行时：pnpm start（端口 3000）
```

#### Dockerfile.nginx

```
1. 分别构建 6 个子应用（VITE_BASE=/micro/xxx/）
2. 每个子应用构建后执行 fixQiankunHtml.mjs 修复 index.html
3. 复制 dist 到 nginx 静态目录
4. 复制 nginx.conf
5. 运行时：nginx -g 'daemon off;'
```

### 7.4 fixQiankunHtml.mjs 的作用

`vite-plugin-qiankun` 生产构建会在 `index.html` 中注入内联 `import('/micro/xxx/static/...js')` 脚本。Qiankun 的 `import-html-entry` 通过 **eval** 执行脚本，无法运行 ES Module 的 `import()`，会导致：

```
application 'app-home' died in status LOADING_SOURCE_CODE:
Failed to execute 'enqueue' on 'ReadableStreamDefaultController'...
```

修复脚本将该内联 `import()` 替换为动态创建 `<script type="module">` 标签的方式，使浏览器原生加载 ES Module。

## 8. 路由与 URL 规划

### 8.1 Nginx 路由规则

| 请求路径 | 处理方式 | 示例 |
|---------|---------|------|
| `/api/*` | 反向代理 → api:3001 | `/api/article/list` |
| `/micro/home/` | 静态文件（alias） | 子应用首页 |
| `/micro/about/` | 静态文件 | 关于我 |
| `/micro/blog/` | 静态文件 | 博客 |
| `/micro/projects/` | 静态文件 | 项目 |
| `/micro/contact/` | 静态文件 | 联系 |
| `/micro/links/` | 静态文件 | 友链 |
| `/micro/home`（无斜杠） | **301 重定向** → `/micro/home/` | 兼容 Qiankun entry |
| `/*` | 反向代理 → web:3000 | `/`、`/about`、`/blog/1` 等 |

### 8.2 API 接口列表

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/article/list` | 文章列表 |
| GET | `/api/article/:id` | 文章详情 |
| GET | `/api/article/slug/:slug` | 按 slug 查文章 |
| GET | `/api/project/list` | 项目列表 |
| POST | `/api/message` | 提交留言 |
| GET | `/api/link/list` | 友链列表 |

### 8.3 验证路由是否正常

```bash
# 主站
curl -I http://你的IP/

# API
curl http://你的IP/api/article/list

# 子应用 entry（应返回子应用 HTML，不是 Next.js 404）
curl -I http://你的IP/micro/home/
curl -I http://你的IP/micro/home     # 应 301 重定向

# 子应用静态 JS
curl -I http://你的IP/micro/home/static/index-*.js
```

## 9. 域名与 HTTPS 配置

当前默认使用 IP + HTTP。若需绑定域名并启用 HTTPS，推荐以下方案。

### 9.1 修改环境变量

```env
# .env
PUBLIC_ORIGIN=https://yourdomain.com
HTTP_PORT=80
```

修改后 **必须重新构建 web 镜像**（NEXT_PUBLIC_* 在构建时写入）：

```bash
docker compose -f docker/docker-compose.yml --env-file .env build --no-cache web
docker compose -f docker/docker-compose.yml --env-file .env up -d web nginx
```

### 9.2 方案 A：Certbot + Nginx（推荐）

在宿主机或单独容器中安装 Certbot，申请 Let's Encrypt 证书，并修改 `docker/nginx.conf` 增加 443 监听与 SSL 配置。

```nginx
server {
    listen 443 ssl;
    server_name yourdomain.com;

    ssl_certificate     /etc/nginx/ssl/fullchain.pem;
    ssl_certificate_key /etc/nginx/ssl/privkey.pem;

    # ... 其余 location 配置与 HTTP 相同
}

server {
    listen 80;
    server_name yourdomain.com;
    return 301 https://$host$request_uri;
}
```

### 9.3 方案 B：阿里云 CDN / SLB 终止 HTTPS

1. 域名解析到服务器 IP
2. 在阿里云 CDN 或负载均衡配置 HTTPS 证书
3. CDN/SLB 回源到服务器 80 端口
4. `.env` 中 `PUBLIC_ORIGIN` 改为 `https://yourdomain.com`

### 9.4 域名解析

在域名 DNS 控制台添加 A 记录：

```
类型: A
主机记录: @（或 www）
记录值: 你的服务器公网 IP
TTL: 600
```

## 10. 日常运维命令

以下命令均在服务器 `/opt/personal-site` 目录下执行。

### 10.1 查看容器状态

```bash
docker compose -f docker/docker-compose.yml ps
```

### 10.2 查看日志

```bash
# 所有服务
docker compose -f docker/docker-compose.yml logs -f

# 单个服务
docker compose -f docker/docker-compose.yml logs -f nginx
docker compose -f docker/docker-compose.yml logs -f web
docker compose -f docker/docker-compose.yml logs -f api
```

### 10.3 重启服务

```bash
# 重启全部
docker compose -f docker/docker-compose.yml restart

# 重启单个
docker compose -f docker/docker-compose.yml restart nginx
docker compose -f docker/docker-compose.yml restart web
docker compose -f docker/docker-compose.yml restart api
```

### 10.4 停止 / 启动

```bash
# 停止（保留数据卷）
docker compose -f docker/docker-compose.yml down

# 启动（不重新构建）
docker compose -f docker/docker-compose.yml --env-file .env up -d
```

### 10.5 进入容器调试

```bash
# 进入 api 容器
docker compose -f docker/docker-compose.yml exec api sh

# 进入 nginx 容器
docker compose -f docker/docker-compose.yml exec nginx sh
```

### 10.6 数据库管理

```bash
# 查看数据库文件（在 api 容器内）
docker compose -f docker/docker-compose.yml exec api ls -la /app/data/

# 手动执行迁移
docker compose -f docker/docker-compose.yml exec api pnpm exec prisma migrate deploy

# 备份数据库
docker cp $(docker compose -f docker/docker-compose.yml ps -q api):/app/data/prod.db ./backup-$(date +%Y%m%d).db
```

### 10.7 磁盘空间清理

Docker 构建会产生大量缓存，定期清理：

```bash
# 清理未使用的镜像、容器、构建缓存

# 仅清理构建缓存
docker builder prune
```

## 12. 故障排查

### 12.1 网站无法访问（连接超时）

| 检查项 | 命令 / 操作 |
|--------|-----------|
| 容器是否运行 | `docker compose ps` |
| 防火墙是否放行 80 | 阿里云控制台 → 防火墙 |
| 端口是否监听 | `ss -tlnp \| grep :80` |
| Nginx 日志 | `docker compose logs nginx` |

### 12.2 页面或 API 返回 502 Bad Gateway

**常见原因**：

| 场景 | 说明 |
|------|------|
| 容器未就绪 | 刚 `up -d --build` 后，Next.js / NestJS 仍在启动，nginx 已对外服务 |
| web 容器崩溃 | 内存不足 OOM，Next.js 进程退出，所有页面（含 `/projects`）均 502 |
| api 容器崩溃 | Prisma 迁移失败或数据库异常，仅 `/api/*` 502 |
| 服务未运行 | 端口 80 连接被拒绝或容器 Exited |

**排查步骤**（在服务器 `/opt/personal-site` 执行）：

```bash
cd /opt/personal-site
docker compose -f docker/docker-compose.yml ps
# 三个容器 nginx / web / api 均应为 Up，web、api 为 healthy

docker compose -f docker/docker-compose.yml logs --tail=50 web
docker compose -f docker/docker-compose.yml logs --tail=50 api
docker compose -f docker/docker-compose.yml logs --tail=30 nginx

# 在服务器本机探测（绕过本地代理）
curl -I http://127.0.0.1/projects
curl -I http://127.0.0.1/api/article/list
```

**快速恢复**：

```bash
cd /opt/personal-site
docker compose -f docker/docker-compose.yml restart web api nginx
# 或完整重建
docker compose -f docker/docker-compose.yml up -d --build
```

**若 web 反复重启（OOM）**：为服务器添加 swap 后重建，见 [12.6](#126-docker-构建失败内存不足)。

**Next.js 构建报错 `two parallel pages that resolve to the same path`**：服务器上仍存在旧的 `next-host/app/about` 等目录（页面已迁到 `app/(site)/`），tar 解压不会自动删除旧文件。`deploy.sh` 已在解压后清理；也可手动执行：

```bash
cd /opt/personal-site
rm -rf next-host/app/about next-host/app/blog next-host/app/contact \
       next-host/app/links next-host/app/projects
rm -f next-host/app/page.tsx
docker compose -f docker/docker-compose.yml up -d --build web
```

**浏览器显示 502 但远程地址为 `127.0.0.1:7897`**：说明请求经过本机代理（如 Clash），可在浏览器关闭代理后重试，或在服务器上用 `curl` 直接验证。

### 12.3 子应用加载失败：`LOADING_SOURCE_CODE` + `ReadableStream`

**原因**：子应用 `index.html` 含有内联 `import()` 脚本，未被 `fixQiankunHtml.mjs` 修复。

**排查**：

```bash
curl -s http://你的IP/micro/home/ | head -10
# 正确：应看到 (function(){ var s=document.createElement('script'); s.type='module'...
# 错误：应看到 import('/micro/home/static/...
```

**解决**：重新构建 nginx 镜像（确保 Dockerfile.nginx 包含 fixQiankunHtml 步骤）。

### 12.4 子应用加载失败：`__next_f.push` 相关错误

**原因**：Qiankun entry URL 无尾部斜杠，请求落入 Next.js 404 页面。

**排查**：

```bash
curl -I http://你的IP/micro/home
# 错误：HTTP 404，X-Powered-By: Next.js
# 正确：HTTP 301 → /micro/home/

curl -sL http://你的IP/micro/home | head -3
# 正确：title 应为 app-home，不是「个人全能站点」
```

**解决**：

1. 确认 `nginx.conf` 有无斜杠 301 重定向规则
2. 确认 `Dockerfile.web` 中 `NEXT_PUBLIC_MICRO_*` 带尾部斜杠
3. 重新构建 web + nginx 容器

### 12.5 页面显示「页面加载中...」不消失

**排查步骤**：

1. 浏览器 F12 → Console，查看 Qiankun 报错
2. 浏览器 F12 → Network，检查 `/micro/home/` 及相关 JS 是否 200
3. 确认 API 可访问：`curl http://你的IP/api/article/list`

### 12.6 Docker 构建失败：内存不足

2 GiB 内存服务器构建 6 个子应用时可能 OOM。

**解决**：

```bash
# 添加 2G swap
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# 然后重新构建
docker compose -f docker/docker-compose.yml --env-file .env build
```

### 12.7 构建失败：`artifacts/api` not found

**原因**：使用了错误的 compose 文件路径（如 `deploy/docker-compose.yml`）。

**解决**：始终使用 `docker/docker-compose.yml`：

```bash
docker compose -f docker/docker-compose.yml --env-file .env up -d --build
```

## 13. 安全建议

1. **修改默认 SSH 密码**，改用密钥登录
2. **不要将密码提交到 Git**；`.env` 已在 `.gitignore` 中
3. **限制 SSH 来源 IP**（阿里云防火墙仅允许你的 IP 访问 22 端口）
4. **定期备份数据库**：

   ```bash
   docker cp $(docker compose -f docker/docker-compose.yml ps -q api):/app/data/prod.db ~/backup.db
   ```

5. **启用 HTTPS**（见第 9 节），避免明文传输
6. **定期更新系统与 Docker**：

   ```bash
   yum update -y   # CentOS / Alibaba Cloud Linux
   docker compose pull  # 更新基础镜像（若改用 pull 方式）
   ```
